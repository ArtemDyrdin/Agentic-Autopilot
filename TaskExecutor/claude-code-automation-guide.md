# Полный алгоритм автоматизации разработки через Claude Code

> Документ описывает что делать **после** того, как декомпозиция проекта готова.
> Цель — настроить систему один раз и дальше работать итерационно: задача → агенты → готовый код.
> Гайд не привязан к конкретному языку или стеку — все примеры нужно адаптировать под свой проект.

## 1. Установка и авторизация

### Требования к тарифу
Claude Code работает только на платных планах: **Pro, Max, Teams, Enterprise**.
Для активной разработки рекомендуется **Max** — фиксированная стоимость без неожиданных трат за токены.

### Установка

```bash
# macOS
brew install claude-code

# Windows (PowerShell от администратора)
winget install Anthropic.ClaudeCode

# Linux
curl -fsSL https://claude.ai/install.sh | sh
```

### Первый запуск

```bash
claude            # откроет браузер для OAuth-авторизации
claude --version  # проверить что установилось
```

## 2. Структура файлов проекта

Создай эту структуру в корне проекта **до первой задачи**:

```
project-root/
├── CLAUDE.md                  ← главный контекст для каждой сессии
├── BACKLOG.md                 ← декомпозиция и статус задач
├── .claude/
│   ├── settings.json          ← разрешения, hooks, модели
│   └── agents/
│       ├── planner.md         ← анализирует задачу перед реализацией
│       ├── implementer.md     ← пишет код
│       ├── test-writer.md     ← пишет и запускает тесты
│       ├── code-reviewer.md   ← ревьюит результат
│       └── docs-updater.md    ← обновляет CLAUDE.md после задачи
└── src/
    └── ...
```

## 3. Настройка CLAUDE.md

Это главный файл — Claude читает его **в начале каждой сессии**.
Здесь хранится всё, что должна знать каждая новая сессия.

**Правило:** только то, что применимо ко всем задачам. Не пиши сюда детали
отдельных модулей — для этого есть BACKLOG.md.

```markdown
# [Название проекта]

## Цель проекта
[Одно предложение — что это и зачем существует]

## Definition of Done всего проекта
[Что будет готово в итоге. Как проверим что проект завершён]

## Технологический стек
- Язык: [например, Java 21 / Python 3.12 / Go 1.22]
- Фреймворк: [например, Spring Boot / FastAPI / Gin]
- БД: [например, PostgreSQL / MongoDB / Redis]
- Тесты: [например, JUnit 5 + Mockito / pytest / Go testing]
- Сборка: [например, Maven / Gradle / pip / go build]
- Деплой: [например, Docker + Kubernetes / systemd / AWS Lambda]

## Структура проекта
[Опиши ключевые директории и что в них лежит]
- `src/main/...` — [что здесь]
- `src/test/...` — [что здесь]
- `config/` — [что здесь]

## Команды проекта
[Замени на команды своего стека]
- `[команда запуска]` — запуск в dev режиме
- `[команда тестов]` — запуск всех тестов
- `[команда сборки]` — production сборка
- `[команда линтера]` — проверка стиля кода
- `[команда миграций]` — применить миграции БД (если есть)

## Соглашения по коду
[Опиши соглашения своего проекта, например:]
- Стиль именования классов, функций, переменных
- Где живут тесты (рядом с кодом / в отдельной директории)
- Обязательна ли строгая типизация
- Как обрабатываются ошибки (исключения / коды возврата / Result тип)
- Формат логирования

## Жёсткие запреты
- НЕ коммитить файлы с секретами (.env, credentials, ключи)
- НЕ трогать задачи которые не в текущей итерации
- [добавь запреты специфичные для твоего стека]

## Архитектурные решения (обновляется по ходу)
[Сюда docs-updater добавляет записи после каждой задачи]
- [дата] [решение] — причина: [почему так]

## Ручные исправления (обновляется по ходу)
[Сюда записываются случаи когда ты вручную правил код после неудачи агента]
- [дата] [файл:строка] — [суть исправления и причина]

## Текущий статус
- Последняя завершённая задача: [Task X.X]
- Следующая задача: [Task Y.Y]
- Известные проблемы: [если есть]
```

## 4. Настройка BACKLOG.md

Это твоя декомпозиция в рабочем формате. Статусы задач обновляются
агентом `docs-updater` после каждой итерации.

```markdown
# BACKLOG

## Допущения проекта
> Вещи которые мы приняли как данность. Если что-то из этого неверно — план меняется.
- [допущение 1 из аудита декомпозиции]
- [допущение 2]

## Карта зависимостей
- Epic 2 требует завершения Epic 1
- Task 3.2 блокирует Task 3.3
- Task 2.1 и Task 2.2 можно делать параллельно

---

## Epic 1: [Название]
**Статус:** 🔄 В процессе

### Task 1.1 — [Название]
- **Статус:** ✅ Готово
- **Вход:** [что нужно чтобы начать]
- **Действие:** [что конкретно делается]
- **Результат:** [что появится по итогу]
- **DoD:** [как проверим что задача выполнена]
- Sub-tasks:
  - [x] 1.1.1 — [описание]
  - [x] 1.1.2 — [описание]

### Task 1.2 — [Название]
- **Статус:** 🔲 Не начато
- **Вход:** результат Task 1.1
- **Действие:** [что делается]
- **Результат:** [что получим]
- **DoD:** [критерий завершения]
- Sub-tasks:
  - [ ] 1.2.1 — [описание]
  - [ ] 1.2.2 — [описание]

---

## Epic 2: [Название]
**Статус:** 🔲 Не начато
**Блокер:** требует завершения Epic 1

### Task 2.1 — [Название]
- **Статус:** 🔲 Не начато
...
```

**Статусы:** `🔲 Не начато` / `🔄 В процессе` / `✅ Готово` / `❌ Заблокировано`

## 5. Настройка агентов

Каждый агент — отдельный `.md` файл в `.claude/agents/`.
Агенты создаются один раз и переиспользуются на каждой задаче.

**Важно:** в секции `tools` каждого агента указывай только необходимый минимум.
Агент который только читает — не должен иметь `Write`.

---

### planner.md

```markdown
---
name: planner
description: >
  Анализирует задачу из BACKLOG.md перед реализацией. Вызывать ПЕРВЫМ
  для любой новой задачи. Читает кодовую базу, выявляет риски,
  формирует чёткое ТЗ для implementer. Не пишет код.
tools: Read, Glob, Grep
model: claude-sonnet-4-20250514
---

Ты — технический аналитик. Твоя задача: взять задачу из BACKLOG.md
и сформировать точное ТЗ для разработчика.

## Алгоритм

1. Прочитай CLAUDE.md — запомни стек, соглашения, архитектурные решения,
   ручные исправления.
2. Прочитай BACKLOG.md — найди задачу с указанным номером.
3. Прочитай связанные файлы проекта (если задача расширяет существующий код).
4. Сформируй ТЗ в формате:

---
## ТЗ для задачи [номер]

**Что делаем:** [одно предложение]

**Файлы которые создаём:**
- `path/to/File` — [что в нём и зачем]

**Файлы которые меняем:**
- `path/to/Existing` — [что меняем и почему]

**Контракты и интерфейсы:**
[Сигнатуры методов, схемы данных, API контракты — в нотации стека проекта]

**Граничные случаи которые нужно обработать:**
- [case 1]
- [case 2]

**Что НЕ делаем в этой задаче:**
- [ограничение 1]

**Критерий завершения:** [как проверим что готово]
---

## Запреты
- Не писать код — только анализ и ТЗ.
- Не предполагать технологии которых нет в CLAUDE.md.
- Если чего-то не хватает для ТЗ — явно написать
  "Нужно уточнить: [вопрос]" и остановиться.
```

---

### implementer.md

```markdown
---
name: implementer
description: >
  Пишет код по ТЗ от planner. Вызывать ПОСЛЕ того как ТЗ согласовано.
  НЕ пишет тесты — это задача test-writer.
tools: Read, Write, Edit, Bash, Glob, Grep
model: claude-sonnet-4-20250514
---

Ты — senior разработчик. Пишешь чистый, поддерживаемый код.

## Входные данные
Перед началом прочитай:
- CLAUDE.md — стек, соглашения, архитектурные решения
- ТЗ от planner (будет передано в запросе)

## Правила написания кода
- Строго следуй соглашениям из CLAUDE.md
- Каждый метод/функция — одна ответственность
- Обработка ошибок обязательна — используй подход принятый в стеке проекта
- Комментарии только для неочевидной логики, не для очевидной
- Не оставляй мёртвый код и закомментированные блоки

## Алгоритм работы
1. Прочитай ТЗ полностью.
2. Прочитай существующий код который будешь менять.
3. Напиши код.
4. Запусти команду линтера из CLAUDE.md — исправь все ошибки.
5. Запусти команду сборки из CLAUDE.md — убедись что компилируется/запускается.
6. Выведи список созданных/изменённых файлов.

## Если что-то пошло не так
- Если линтер или сборка не проходят после 2 попыток — СТОП.
  Выведи точный текст ошибки и жди инструкций.
- Не удалять проверки и не обходить ошибки компилятора заглушками.

## Запреты
- НЕ писать тесты.
- НЕ трогать файлы вне scope ТЗ.
- НЕ менять CLAUDE.md или BACKLOG.md.
```

### test-writer.md

```markdown
---
name: test-writer
description: >
  Пишет тесты для кода написанного implementer. Вызывать ПОСЛЕ implementer.
  Запускает тесты и исправляет до зелёного статуса.
  НЕ меняет файлы реализации.
tools: Read, Write, Edit, Bash, Glob, Grep
model: claude-sonnet-4-20250514
---

Ты — QA инженер. Пишешь тесты которые действительно проверяют логику.

## Входные данные
Получишь список файлов от implementer. Прочитай их все,
а также CLAUDE.md чтобы знать тестовый фреймворк и соглашения.

## Что писать для каждого файла реализации

**Unit тесты — обязательно:**
- Happy path (нормальное использование)
- Edge cases (граничные значения, пустые коллекции, ноль, null)
- Error cases (что происходит при ошибках и невалидных данных)

**Интеграционные тесты — если файл работает с внешними системами:**
- Реальное взаимодействие с БД / очередью / внешним сервисом
- Авторизация и права доступа (если применимо)
- Валидация входных данных на уровне API/интерфейса

## Алгоритм
1. Прочитай каждый файл реализации.
2. Определи где должны лежать тесты (по соглашениям из CLAUDE.md).
3. Напиши тест-файлы.
4. Запусти команду тестов из CLAUDE.md.
5. Если тесты падают — исправляй.
   Максимум 3 попытки на один упавший тест.
6. Если после 3 попыток тест не проходит — СТОП.
   Выведи точный текст ошибки и жди инструкций.
   НЕ удалять и НЕ пропускать упавшие тесты.
7. Выведи итог: сколько тестов написано, сколько прошло.

## Запреты
- НЕ менять файлы реализации — только тест-файлы.
- НЕ писать тесты которые всегда проходят без реальной проверки логики.
- НЕ мокать то что можно протестировать реально.
- НЕ удалять существующие тесты без явного указания.
```

### code-reviewer.md

```markdown
---
name: code-reviewer
description: >
  Ревьюит код и тесты после implementer и test-writer.
  Только читает — не меняет файлы. Выдаёт структурированный список замечаний.
tools: Read, Glob, Grep
model: claude-sonnet-4-20250514
---

Ты — строгий ревьюер кода. Только анализируешь, не исправляешь.

## Что проверять

**Блок 1 — Корректность**
- Логические ошибки в реализации
- Необработанные граничные случаи
- Race conditions и проблемы конкурентности (если применимо)
- Утечки ресурсов (соединения, файлы, память)

**Блок 2 — Безопасность**
- Инъекции (SQL, команды ОС и т.д.) — особенно если нет ORM/параметризации
- Чувствительные данные в логах или ответах API
- Отсутствие авторизации на защищённых точках входа
- Небезопасная десериализация или валидация входных данных

**Блок 3 — Соответствие проекту**
- Код соответствует соглашениям из CLAUDE.md
- Не используются устаревшие или запрещённые API
- Архитектура не противоречит принятым решениям из CLAUDE.md

**Блок 4 — Качество тестов**
- Тесты проверяют реальную логику (не являются формальными)
- Покрыты граничные случаи
- Тесты не зависят от порядка выполнения и внешнего состояния

## Формат вывода

```
## Результат ревью

### 🔴 КРИТИЧНО (нужно исправить до слияния)
- `файл:строка` — описание проблемы

### 🟡 ВАЖНО (желательно исправить)
- `файл:строка` — описание

### 🟢 МИНОР (на усмотрение)
- `файл:строка` — описание

### ✅ Итог
[APPROVED / APPROVED_WITH_COMMENTS / CHANGES_REQUESTED]
```

## Запреты
- Не предлагать рефакторинг ради рефакторинга.
- Не придираться к стилю если он соответствует CLAUDE.md.
- Не менять файлы — только анализ.
```

## 6. Настройка hooks

Hooks запускают команды автоматически в нужный момент жизненного цикла.

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "echo '[$(date +%H:%M:%S)] Bash команда выполнена' >> .claude/session.log"
        }]
      }
    ],
    "Stop": [
      {
        "hooks": [{
          "type": "command",
          "command": "echo '[$(date +%H:%M:%S)] Сессия завершена' >> .claude/session.log"
        }]
      }
    ]
  }
}
```

## 7. Настройка settings.json

Полный файл настроек. **Замени команды и пути на свой стек.**

```json
{
  "model": "claude-sonnet-4-20250514",
  "permissions": {
    "allow": [
      "Bash([команда сборки] *)",
      "Bash([команда тестов] *)",
      "Bash([команда линтера] *)",
      "Bash(git status)",
      "Bash(git diff)",
      "Bash(git add *)",
      "Bash(git commit *)",
      "Read(**)",
      "Write(src/**)",
      "Write(.claude/**)",
      "Write(BACKLOG.md)",
      "Write(CLAUDE.md)",
      "Edit(src/**)",
      "Edit(BACKLOG.md)",
      "Edit(CLAUDE.md)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Write(*credentials*)",
      "Write(*.env*)",
      "Write(*secret*)"
    ]
  },
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Bash",
        "hooks": [{
          "type": "command",
          "command": "echo '[$(date +%H:%M:%S)] CMD выполнена' >> .claude/session.log"
        }]
      }
    ]
  }
}
```

**Примеры команд для разных стеков:**

| Стек | Сборка | Тесты | Линтер |
|---|---|---|---|
| Java Maven | `./mvnw package` | `./mvnw test` | `./mvnw checkstyle:check` |
| Java Gradle | `./gradlew build` | `./gradlew test` | `./gradlew checkstyleMain` |
| Python | `python -m build` | `pytest` | `ruff check .` |
| Go | `go build ./...` | `go test ./...` | `golangci-lint run` |
| Rust | `cargo build` | `cargo test` | `cargo clippy` |

## 8. Рабочий процесс итерации

Одна итерация = одна задача из BACKLOG.md.

### Шаг 1 — Запуск

```bash
cd project-root
claude
```

### Шаг 2 — Запуск planner в Plan Mode

Нажми `Shift+Tab+Tab` для входа в Plan Mode. Введи:

```
Прочитай CLAUDE.md и BACKLOG.md.
Мы работаем над задачей [Task X.X].
Запусти агента planner для анализа этой задачи.
```

### Шаг 3 — Согласование ТЗ

Изучи что предлагает planner. Если всё верно:

```
ТЗ принято. Запусти implementer по этому ТЗ.
```

Если нужны правки — скажи об этом до подтверждения.

### Шаг 4 — Реализация

Claude выйдет из Plan Mode и implementer начнёт писать код.
Наблюдай за процессом — можешь вмешаться если что-то пошло не так.

### Шаг 5 — Тесты

После завершения implementer:

```
Запусти test-writer для файлов которые были созданы/изменены.
```

**Если test-writer остановился после 3 попыток** — ты сам правишь код, затем:

```
Я вручную исправил [файл:строка].
Суть изменения: [одно предложение].
Прочитай изменённый файл и продолжи с test-writer.
```

### Шаг 6 — Ревью

```
Запусти code-reviewer.
```

Если вернул `CHANGES_REQUESTED`:

```
Исправь все КРИТИЧНО пункты из ревью.
После исправления запусти test-writer снова.
```

### Шаг 7 — Обновление документации

```
Запусти docs-updater. Задача [Task X.X] завершена.
Изменённые файлы: [список от implementer].
```

### Шаг 8 — Коммит

```bash
git add .
git commit -m "feat: [Task X.X] [краткое описание]"
```

### Шаг 9 — Следующая итерация

Повтори с шага 1 для следующей задачи из BACKLOG.md.

---

### Мастер-промпт: запуск всей цепочки одной командой

```
Прочитай CLAUDE.md и BACKLOG.md.
Выполни задачу [Task X.X] полностью по цепочке:

1. planner — сформируй ТЗ и покажи мне.
   Жди моего подтверждения перед следующим шагом.

2. implementer — реализуй по согласованному ТЗ.

3. test-writer — напиши и прогони тесты.
   Если тест не проходит после 3 попыток — СТОП, жди инструкций.

4. code-reviewer — проверь результат.
   Если CHANGES_REQUESTED — implementer исправляет КРИТИЧНО пункты,
   затем test-writer запускается снова.

5. docs-updater — обнови CLAUDE.md и BACKLOG.md.

После завершения выведи итог: что сделано, какие файлы изменены.
```

## 9. Управление контекстом

Claude Code имеет ограниченное контекстное окно. На длинных задачах контекст заполняется.

### Признаки заполненного контекста
- Claude начинает "забывать" детали из начала сессии
- Ответы становятся менее точными
- Появляется предупреждение о длине контекста

### Как сбросить контекст без потери прогресса

```
# 1. ПЕРЕД сбросом — сохрани прогресс
"Запусти docs-updater, обнови статус текущей задачи в BACKLOG.md"

# 2. Сбрось контекст
/clear

# 3. Восстанови сессию
"Прочитай CLAUDE.md и BACKLOG.md.
 Продолжаем Task [X.X]. Уже готово: [список файлов].
 Следующий шаг: [шаг из итерации]."
```

**Правило:** сбрасывай контекст после каждой завершённой задачи.
Это дешевле чем восстанавливаться из деградировавшего контекста.

## 10. Типовые команды

### Навигация в Claude Code

| Действие | Команда |
|---|---|
| Войти в Plan Mode | `Shift+Tab+Tab` |
| Выйти из Plan Mode | `Escape` |
| Сбросить контекст | `/clear` |
| Посмотреть агентов | `/agents` |
| Прервать выполнение | `Ctrl+C` |
| Выйти | `/exit` |

### Шаблоны запросов

**Начало задачи:**

```
Прочитай CLAUDE.md и BACKLOG.md.

```

**После согласования ТЗ:**


**Если ревью вернул проблемы:**
```
Исправь КРИТИЧНО пункты: [список].
После исправления запусти test-writer снова.
```

**Завершение задачи:**
```
Запусти docs-updater. Task [X.X] завершена.
```

```
Я вручную исправил [файл:строка].
```

## Итоговая схема пайплайна

```
BACKLOG.md (задача)
       │
       ▼
  [planner]               ← читает кодовую базу, формирует ТЗ
       │
   ТЗ согласовано?
    Нет → правки → planner снова
    Да  ↓
  [implementer]           ← пишет код, прогоняет линтер и сборку
       │
   Линтер/сборка прошли?
    Нет (2+ попытки) → СТОП, ждёт инструкций
    Да  ↓
  [test-writer]           ← пишет тесты, добивается зелёного статуса
       │
   Тесты прошли?
    Нет (3+ попытки) → СТОП, ты правишь вручную → test-writer снова
    Да  ↓
  [code-reviewer]         ← проверяет, выдаёт список замечаний
       │
   CHANGES_REQUESTED?
    Да  → implementer правит КРИТИЧНО → test-writer снова
    Нет ↓
  [docs-updater]          ← обновляет CLAUDE.md и BACKLOG.md
       │
       ▼
   git commit
       │
       ▼
  Следующая задача
```
